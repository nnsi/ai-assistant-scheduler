# 日記を書く

セッション中の開発内容、反省点、学びを記録する。

## 手順

1. 現在の日付を取得する

```bash
# Linux
date +%Y%m%d

# Windows
Get-Date -Format "yyyyMMdd"
```

2. `/docs/diary/YYYYMMDD.md` を作成または追記する

3. `/docs/diary/CLAUDE.md` のガイドラインに従って記述する

## 記述のポイント

### 必ず含めるべき内容

- **今日やったこと**: 実装した機能、修正したバグ、調査した内容
- **反省点/自己批判**: 「こうすべきだった」「見落としていた」など
- **良かった点**: うまくいったこと、ユーザーから得た知見
- **技術メモ**: 今後役立ちそうな技術的な気づき

### 書き方の原則（CLAUDE.mdより）

- ユーザーに迎合しない。本当に思ったことを正直に書く
- 異論があれば書く。ユーザーの判断に従ったが自分は違う意見だった場合
- 自己批判も隠さない。「言えなかった」「気づかなかった」など

### 避けるべきこと

- ユーザーを持ち上げるだけの記述
- 実際には感じていないポジティブな感想
- 反省点のない「うまくいきました」報告

## 日記の活用

日記は「書いて終わり」ではなく、読み返して改善に活かす。

### 振り返りセッションで行うこと

1. 過去の日記を読み返す
2. 繰り返し出てくるパターンを抽出する
3. 以下に反映する：
   - **CLAUDE.md**: プロジェクト全体で守るべきルール
   - **スキル**: 手順化できる繰り返しタスク
   - **サブエージェント**: 並列で実行できる専門タスク

### 例：日記から抽出されたパターン

| 日記での記述 | 反映先 |
|---|---|
| 「テストDBスキーマの同期を忘れた」が3回 | CLAUDE.md |
| 「サブエージェントの誤検出を検証せずに鵜呑みにした」 | code-review, security-audit スキル |
| 「React 18 Strict Modeで二重実行された」 | CLAUDE.md |
